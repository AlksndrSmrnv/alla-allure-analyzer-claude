# Инструкция пользователя alla

## Что такое alla

`alla` — помощник по разбору упавших автотестов в Allure TestOps. После
прогона она смотрит на активные failed/broken падения, группирует похожие
ошибки в кластеры, сверяет их с базой знаний проекта и показывает, что
вероятнее всего сломалось и что делать дальше.

Alla не анализирует muted-падения как активную работу: они считаются в
статистике отдельно, но не попадают в кластеры. Retry/hidden результаты тоже
исключаются, чтобы отчёт опирался на финальные результаты тестов.

## Что появляется после анализа

Обычно после работы alla есть:

1. Ссылка на HTML-отчёт в карточке запуска TestOps, если серверу известен
   публичный URL отчёта.
2. Комментарии под тест-кейсами в TestOps с префиксом `[alla]`, если включена
   запись результатов в TestOps.
3. Сам HTML-отчёт: главный экран для разбора кластеров и обучения базы знаний.

Если администратор запустил анализ с `push_to_testops=false`, отчёт может быть
сформирован без новых комментариев и без записи ссылки в TestOps.

## Как читать HTML-отчёт

Вверху отчёта находятся:

- ссылка на запуск в TestOps;
- карточки статистики: всего, успешно, упало, сломано, пропущено, кластеров;
- onboarding-блок, если база знаний проекта ещё пустая или не настроена;
- раздел **Итоговый анализ прогона**, если включён LLM summary.

Дальше идёт раздел **Кластеры падений**. Кластер — группа тестов, которые
похожи по сообщению ошибки, стеку, логу приложения и шагу теста.

Внутри кластера могут быть:

- **AI Анализ** — гипотеза и шаги исправления от LLM;
- **Шаг теста** — breadcrumb упавшего execution-шага;
- **Пример ошибки** — сообщение ошибки, корреляция и/или фрагмент лога;
- **База знаний** или **Знания проекта** — найденные известные причины;
- **Создать решение для кластера** — форма добавления записи в базу знаний;
- **Затронутые тесты** — ссылки на конкретные тесты.

Если проект новый, глобальные стартовые подсказки могут быть спрятаны под
кнопкой **Показать starter pack**. Это справочные примеры; проектное решение
лучше сохранить отдельно через форму кластера.

## Наполнение базы знаний

Главная польза alla появляется, когда команда сохраняет подтверждённые причины
и решения. Тогда повторные падения сразу получают проверенную рекомендацию.

### Создать решение для кластера

Нажмите **Создать решение для кластера**. В форме уже подставлены данные из
текущего кластера: заголовок, категория, пример ошибки и иногда шаг теста.

Основное поле — **Шаги по устранению**. Запишите туда:

- что именно сломалось;
- почему это происходит;
- какие конкретные действия исправляют проблему.

Поля **Заголовок**, **Категория**, **Пример ошибки** и **Описание** можно
поправить перед сохранением. Если чекбокс шага теста включён, запись становится
step-aware: она будет срабатывать только на совместимые шаги, что снижает
ложные совпадения.

После нажатия **Сохранить в проект** запись попадает в базу знаний этого
проекта и начнёт учитываться в следующих анализах.

### Скопировать starter pack

Если в новом проекте подошла глобальная подсказка из starter pack, откройте
её и нажмите **Скопировать в проект**. После копирования запись можно
доработать под конкретный проект.

### Обновить существующую запись

У найденной записи нажмите **Обновить запись в базе знаний**. Можно изменить
заголовок, описание, категорию, пример ошибки, шаг теста и шаги устранения.

Кнопка **Обновить из кластера** рядом с примером ошибки подставляет актуальный
пример из текущего кластера.

## Фидбек по совпадениям

У совпадений с базой знаний есть кнопки:

- **👍** — совпадение полезное;
- **👎** — запись не подходит к этой ошибке.

Голос сохраняется по стабильной сигнатуре ошибки. Если команда поставила
лайк, такое совпадение в будущем может отображаться как **Ранее подтверждено**
и подниматься выше обычных текстовых совпадений. Дизлайк скрывает неподходящую
запись для той же сигнатуры.

## Объединение кластеров

Если одна причина разнесена на несколько кластеров:

1. Отметьте нужные кластеры чекбоксом **Выбрать**.
2. В панели снизу нажмите **Объединить выбранные**.
3. Alla сохранит merge rule для проекта.

На следующих анализах кластеры с теми же стабильными сигнатурами будут
объединяться автоматически. Кнопка **Сбросить** очищает выделение.

Чекбоксы появляются только когда включён сервер интерактивности и для кластера
можно построить стабильную сигнатуру.

## Перезапуск анализа из отчёта

Если отчёт открыт с сервера alla, вверху может быть кнопка
**Перезапустить анализ**. Она запускает новый HTML-анализ этого же запуска с
`push_to_testops=false`: страница обновится, но новые комментарии в TestOps не
будут записаны.

Полный перезапуск с записью комментариев обычно делает администратор или CI.
Перед таким перезапуском старые комментарии `[alla]` можно удалить отдельной
командой, чтобы не плодить дубли.

## Если что-то выглядит не так

- **Нет ссылки на HTML-отчёт в TestOps** — проверьте у администратора, задан ли
  публичный URL отчётов (`ALLURE_SERVER_EXTERNAL_URL` или `ALLURE_REPORT_URL`) и
  разрешён ли push в TestOps.
- **Нет кнопок создания, редактирования, 👍/👎 или объединения** — интерактивный
  режим не подключён. Нужны PostgreSQL база знаний и `ALLURE_FEEDBACK_SERVER_URL`.
- **Нет блока “Итоговый анализ прогона”** — LLM summary не запускался или
  GigaChat не настроен.
- **Подсказка из базы знаний не подходит** — нажмите 👎 и, если причина уже
  понятна, создайте правильную запись для проекта.
- **Один кластер содержит разные причины** — добавьте более точные записи в
  базу знаний или разделите работу при разборе вручную; если проблема
  повторяется наоборот как несколько кластеров, используйте объединение.
- **Отчёт есть, но комментариев в TestOps нет** — анализ мог идти в read-only
  режиме (`push_to_testops=false`) или LLM/база знаний не дали результата для
  push.
