# Launch summary

Гайд для написания итогового отчёта по прогону (`launch_summary` в
payload).

## Источник промпта

```bash
python alla-skill/scripts/get_summary_context.py --run-id $RUN_ID
```

Stdout содержит:

* `system_prompt` + `user_prompt` — готовый промпт (тот же, что
  GigaChat-путь).
* `context.counters` — счётчики прогона.
* `context.top_clusters` — top-10 кластеров по убыванию size с
  meta-данными.
* `context.analyses_used` — сколько per-cluster анализов уже сохранено
  в БД и подмешано в промпт.

Применяй `system_prompt` + `user_prompt` буквально — он содержит
все данные и инструкцию.

## Что должно быть в `summary_text`

Промпт говорит «2–4 абзаца», три раздела:

### 1. Общая картина

* Сколько тестов всего, сколько упало (active failures, без muted).
* Сколько уникальных проблем (кластеров).
* Если есть `unanalyzed_tail` — обязательно упомянуть.

### 2. Ключевые проблемы

Для каждого крупного кластера — одно предложение: что упало, почему,
категория (тест / приложение / окружение / данные). Расставь по
убыванию критичности и количества тестов.

### 3. Приоритетные исправления

1–3 конкретных действия на ближайшее время. Пиши именно
**исправления**, а не диагностику. Привязывай к конкретному
кластеру / сервису / фичечной зоне.

## Стиль

* Краткий, technical, на языке пользователя (по умолчанию русский).
* Без эмодзи.
* Без повторов между абзацами.
* Не упоминай то, чего нет в данных (сервисы, классы, конфиги).

## Полный payload `launch_summary`

В JSON, который ты потом скармливаешь `submit_analysis`:

```json
"launch_summary": {
  "summary_text": "Прогон #12345 — 38 активных падений в 320 тестах ...",
  "key_findings": [
    "Кластер c-abc — массовые ConnectionTimeout на /api/v1/orders.",
    "Кластер c-def — flaky локатор Pay-кнопки в PaymentPage."
  ],
  "priority_actions": [
    "Перезапустить service-billing в стейдже",
    "Стабилизировать селектор Pay в PaymentPage.kt"
  ],
  "unanalyzed_tail": {
    "cluster_count": 0,
    "test_count": 0,
    "note": null
  }
}
```

Поля `key_findings` и `priority_actions` — bullet-списки (3–5
пунктов в каждом). Они подмешиваются в HTML-отчёт под основным
`summary_text`.

`unanalyzed_tail` заполняется только в режиме `>30 кластеров`
(см. `delegation_strategy.md`). Иначе оставь нули и `null`.
