# Анализ одного кластера

Гайд для агента/субагента, анализирующего ОДИН кластер. Единый формат
для Claude / qwen / codex.

## Источник промпта

Контекст и готовый промпт получаешь так:

```bash
python alla-skill/scripts/get_cluster_context.py --run-id $RUN_ID --cluster-id $CID
```

Stdout — JSON с полями:

* `system_prompt` — системный промпт.
* `user_prompt` — пользовательский промпт (готовый, с правилами анализа).
* `context` — структурированные данные кластера (для справки, не вместо
  промпта):
    * `representative` — лог, трейс, message представителя.
    * `members` — все упавшие тесты в кластере с ссылками.
    * `kb_matches` — top-N совпадений с базой знаний (с tier, score,
      описанием, resolution_steps).
    * `kb_query_provenance` — длины частей запроса, по которым
      найдено совпадение.
    * `feedback_context` — данные exact-feedback (для UI).

**Применяй `system_prompt` + `user_prompt` буквально.** Они уже
содержат:

* Роль и правила поведения.
* Все данные (message, trace, log, KB-совпадения).
* Подробную инструкцию по проверке применимости KB.
* Точный требуемый формат ответа (ЧТО СЛОМАЛОСЬ / ПРИЧИНА / КАК
  ИСПРАВИТЬ).

## Правила анализа

Промпт уже содержит подробные правила. Ключевые тезисы для контроля:

### Приоритет источников

1. message / trace / log — первичные факты.
2. KB — только при доказанной применимости (Tier 1 / feedback_exact —
   сразу основная причина; Tier 2 / Tier 3 — нужна цитата фрагмента
   из `error_example`, дословно или близко присутствующего в данных
   кластера).
3. score сам по себе — не доказательство применимости.

### Категории

* `test` — нестабильный код теста (flaky locator, race condition в
  тесте, кривой ассерт).
* `service` — баг в продукте (NullPointer в бэкенде, неправильный
  ответ API).
* `env` — инфра (502 от прокси, недоступная стейдж-БД, разрыв сети).
* `data` — тестовые данные (отсутствует фикстура, истёк токен в
  файле, дубликат записи).

### Длина

`analysis_text`: 200–1500 символов. Без воды. Каждый шаг
"КАК ИСПРАВИТЬ" — конкретный (что/где/как), а не «проверить» / «уточнить».

### Запрет фантазий

Не упоминай сервисы, классы, методы, конфиги, команды, которых нет в
данных. Если данных мало — прямо напиши «данных недостаточно для
определения причины», категория ставится по самой вероятной из
доступных, confidence = `low`.

## Формат возврата

Возвращай **СТРОГО** JSON одного кластера, без markdown-обёртки:

```json
{
  "cluster_id": "c-abc123",
  "category": "service",
  "confidence": "high",
  "analysis_text": "ЧТО СЛОМАЛОСЬ: ...\n\nПРИЧИНА: ...\n\nКАК ИСПРАВИТЬ:\n1. ...\n2. ...\n3. ...",
  "kb_alignment": {
    "matched_kb_entry_ids": [17],
    "rejected_kb_entry_ids": [],
    "rejection_reason": null
  },
  "recommendations": [
    "Перезапустить service-billing pod в стейдже",
    "Проверить health-check на /api/v1/orders"
  ]
}
```

Для batched subagent (2–3 кластера) возвращай массив таких объектов.

## kb_alignment

* `matched_kb_entry_ids` — `entry_id` записей KB (из
  `context.kb_matches[*].entry_id`), которые ты признал применимыми.
* `rejected_kb_entry_ids` — записи, которые ты явно отверг (важно
  для feedback). Передавай `entry_id` всех `kb_matches`, для которых
  ты в `analysis_text` написал «не подходит».
* `rejection_reason` — короткий текст, почему отвергнут (если есть).

## Tail (категория `unanalyzed`)

Только в режиме `>30 кластеров`, для кластеров вне top-30. Никаких
вычислений и текстовых рассуждений — пиши:

```json
{
  "cluster_id": "c-zzz",
  "category": "unanalyzed",
  "confidence": "low",
  "analysis_text": "tail (size=4)",
  "kb_alignment": {"matched_kb_entry_ids": [],
                   "rejected_kb_entry_ids": [],
                   "rejection_reason": null},
  "recommendations": []
}
```

Это исключает кластер из push'а в TestOps и не показывает «не
проанализирован» в HTML — вместо этого помечает как осознанно
пропущенный.
